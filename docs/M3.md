# M3 — network interop, extras, packaging (the final parity milestone)

This file is the kickoff brief for the M3 work session. Read `CLAUDE.md`
first (architecture rules, commands, gotchas), then `README.md` (user-facing
state) and `CREDITS.md` (EQTool source pinned at commit `fdd3f25a`).
M0–M2 are complete: 475 tests, full local feature parity with EQTool.
`git log --oneline` tells the story milestone by milestone.

## Goal

Ship nParse+ 1.0: connect it to the live PigParse network, add the Night
Vision fix and a self-updater, and package it as a macOS .app/DMG (plus
Windows/Linux CI builds) that a non-technical P99 player can install.

## 1. PigParse network interop (the big one)

EQTool users share data through two services; interoperating with them was an
explicit product decision (fallbacks exist, but live interop is the goal):

- **SignalR hub** `https://www.pigparse.org/PP` — real-time. Client→server
  methods: `JoinServerGroup(server)`, `PlayerLocationEvent(player)`,
  `DragonRoarEvent(...)`. Server→client callbacks: `PlayerLocationEvent`,
  `PlayerDisconnected`, `AddCustomTrigger`, `DragonRoarEvent`.
  Reference implementation: EQTool's `Services/SignalrPlayerHub.cs`; DTOs in
  `EQToolShared/HubModels/` (`SignalrPlayerV2`, `SignalRDragonRoar`).
- **REST API** `https://pigparse.azurewebsites.net` — `api/item/wiki`,
  `api/item/postmultiple`, `api/player/getbynames|upsertplayers`,
  `api/zone/npcactivity`, `api/zone/quakev2/{server}`,
  `api/boat/seen|serverActivity/{server}`, `api/rolltimer/timers/{server}`.
  Reference: `EQToolApis/` controllers + EQTool's `Services/PigParseApi.cs`.

Build order that de-risks it:

1. **Probe first.** Write `tools/probe_pigparse.py`: connect with
   `signalrcore` (already a dependency), join a server group, log every frame
   for a few minutes. Commit a sanitized transcript to `tools/`. The wire
   details to confirm: JSON hub protocol, property casing (expect PascalCase),
   enums as ints (`Server`: Green=0 Blue=1 Red=2 Quarm=3 — already in
   `core/enums.py`, values must not change), reconnect/negotiate behavior.
   If `signalrcore` can't speak to the hub, options are vendoring a minimal
   JSON-hub-protocol websocket client or fixing signalrcore — decide from the
   probe evidence.
2. `net/pigparse_hub.py` — client on its own thread, auto-reconnect with
   jittered backoff, `JoinServerGroup` after **every** (re)connect. Outbound:
   location resend every 10 s unless idle >5 min (`pipeline.last_you_activity`
   exists for this), cleared on CampEvent; DragonRoar dedupe window 4 s.
   Never echo your own player back into the UI. Inbound: validate payloads
   into the DTOs (pydantic, PascalCase aliases, `extra="ignore"`) and publish
   the ALREADY-DEFINED remote events in `core/events.py`
   (`OtherPlayerLocationReceivedRemoteEvent`, `PlayerDisconnectReceivedRemoteEvent`,
   `DragonRoarRemoteEvent`). All failures non-fatal; expose a status string
   for the tray.
3. `net/pigparse_api.py` — httpx, 5 s timeout, one retry, `None` on failure.
   Feeds MobInfo enrichment (wiki/pricing), boat rows, quake state, roll
   timers, npcactivity for the maps NPC panel.
4. **Wire the sends**: `grep -rn "TODO(M3)" src/` — every handler that should
   share data (quake, boat sighting, FTE guild, con enrichment, Kael engage)
   has a marked stub with the C# reference named.
5. **Map dots**: the maps window consumes the remote player events (via the
   Qt bridge) to draw other players with name labels and staleness expiry;
   `sharing.mode` setting gates everything (`pigparse` | `nparse` | `off`).
6. `net/nparse_ws.py` — port of legacy `helpers/location_service.py`
   (sheeplauncher websocket protocol, `locationserver/` in-repo is the
   self-hostable server) as the `nparse` mode. Retire the legacy module.
7. Per-character sharing prefs exist on `PlayerInfo` (map_location_sharing,
   share_timers) — honor them.

Tests: offline only — recorded/synthetic hub frames and REST payloads as
fixtures; DTO round-trips; reconnect state machine with a fake transport.
The live probe is a tool, not a test.

## 2. Night Vision fix

`data/visionfix.zip` (already shipped) extracts over the EQ install
(`settings.general.eq_install_dir`). Port EQTool's apply/revert
(`SettingsGeneral.xaml.cs` ~1215-1225): preflight that the dir contains
`eqgame.exe` and `uifiles/`, back up every file that will be overwritten
(to a `visionfix_backup/` beside them), provide Revert. Surface as buttons in
the existing Preferences window (`ui/preferences.py`). Works fine through the
wine wrapper's filesystem — it's just files.

## 3. Self-updater

`updater.py`: check `api.github.com/repos/<owner>/nparseplus/releases/latest`
(the repo has no `origin` remote yet — the owner/repo comes from creating the
GitHub repo; make it a constant), compare with `packaging.version` against
`nparseplus.__version__`, notify via tray. "Install": macOS downloads the DMG
to ~/Downloads and opens it; Windows downloads the zip; Linux opens the
release page. No in-place swap in v1. Respect `general.update_check`.

## 4. Packaging (macOS first)

- `packaging/nparseplus.spec`: PyInstaller **onedir** .app. Two data roots to
  bundle: `src/nparseplus/data/` (package data) AND the repo-root `data/`
  (maps/spells/fonts/ui used by the legacy modules via CWD-relative paths —
  `app._ensure_data_cwd()` and `helpers.resource_path()` handle _MEIPASS;
  verify both paths resolve inside the bundle, this is the likeliest breakage).
  Exclude QtWebEngine unless the Discord overlay is kept (it's ~200 MB; make
  Discord an optional extra and confirm with the user).
- `icon.icns` from `data/ui/icon.png`; `dmgbuild` config for a
  drag-to-Applications DMG; ad-hoc codesign; README gets the
  `xattr -dr com.apple.quarantine` note (no notarization in 1.0).
- `.github/workflows/release.yml`: on tag — macos-14 (arm64 DMG),
  windows-latest (zip), ubuntu (tarball) — attach to a GitHub release.
- Acceptance: the .app launches on this Mac from Finder with no terminal,
  tails the real log dir, and every window works.

## Definition of done

- Full suite green (offline); new net/ code covered by fixture tests.
- Live check: with sharing on, other EQTool users' dots appear on the map
  while playing, and a location you share is visible to EQTool (ask a
  guildmate or run EQTool under wine side by side).
- VisionFix apply+revert verified against the real install (backup restored
  byte-identical).
- Packaged .app passes the acceptance run above.
- README/CLAUDE.md updated (install section points at the release DMG).

## Risks / notes carried over

- `signalrcore` is the biggest unknown — probe before building on it.
- Rate-limit courtesy: EQTool sends location at ~10 s cadence; do not exceed.
- The demo harness (fake log + `NPARSEPLUS_SETTINGS`) is documented in
  CLAUDE.md and is the fastest way to test map dots with synthetic remote
  players (publish remote events on the bus directly).
- Post-1.0 parking lot: P99 login middleman (explicitly skipped), Discord
  auth/auction APIs, inventory watcher, notarization, 3D map view (stretch,
  design notes in the git history and task list).
